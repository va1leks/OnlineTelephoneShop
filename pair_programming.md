# Парное программирование
## 1. Описание выполняемой задачи
В рамках проекта разрабатывалась фронтенд-часть для REST API сервиса онлайн-магазина телефонов. Приложение работает с четырьмя основными моделями данных:
- **Product** — продукт (телефоны и товары)
- **Order** — заказ
- **User** — пользователь
- **Cart** — корзина
  Во время выполнения задания были опробованы два метода парного программирования:
- «на равных»
- «пинг-понг»
---
## 2. Сравнение применяемых стилей
По итогам работы более удобным оказался стиль **«на равных»**.
В этом формате оба участника активно участвуют в принятии решений, обсуждают архитектуру и по очереди меняются ролями. Смена «водителя» происходила после завершения логических частей кода, что было естественно и удобно.
Стиль **«пинг-понг»** оказался полезным при работе по принципу TDD (сначала тест — потом реализация). Однако на практике он замедлял процесс, так как требовал постоянного написания тестов и строгого соблюдения очередности ролей. Из-за непривычности такого формата темп разработки снижался.
---
## 3. Положительные результаты
В процессе работы были получены следующие преимущества:
- **Повышение качества кода.** Многие ошибки выявлялись сразу во время написания.
- **Обмен опытом.** Узнали новые горячие клавиши, инструменты и подходы к организации компонентов.
- **Более продуманная архитектура.** Постоянное обсуждение работы с моделями Order и User помогло избежать дублирования кода.
- **Лучшая концентрация.** Работая в паре, было меньше отвлечений на посторонние факторы.
- **Общее понимание системы.** Лучше разобрались в структуре приложения и взаимодействии его частей.
---
## 4. Возникшие трудности и способы их решения
Во время работы появились некоторые сложности:
### Разный темп работы
Участники работали с разной скоростью, что иногда вызывало напряжение.
**Решение:** заранее договариваться о формате сессии и делать короткие паузы для синхронизации.
### Различия в стиле кода
Были разные привычки в именовании переменных и форматировании.
**Решение:** использовать единые инструменты форматирования (например, ESLint и Prettier).
### Снижение вовлечённости при «пинг-понге»
Иногда участник, который не писал код, становился менее активным.
**Решение:** договориться обязательно комментировать действия и задавать вопросы во время работы.
---
## 5. Перспективы применения парного программирования
Парное программирование целесообразно использовать в следующих случаях:
- при проектировании архитектуры;
- при реализации сложной логики;
- при рефакторинге;
- при поиске и исправлении ошибок в незнакомом коде;
- при проведении ревью в реальном времени.
  Для простых и рутинных задач индивидуальная работа может быть более эффективной.
---
## Вывод
Парное программирование показало себя как полезный инструмент командной разработки. Несмотря на некоторые организационные сложности, оно помогает улучшить качество кода, ускорить обмен знаниями и повысить вовлечённость участников в процесс разработки.
