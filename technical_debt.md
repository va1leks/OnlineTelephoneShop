# Технический долг

# 1. Анализ проекта: Выявление технического долга

## 1.1 Неясный / Трудночитаемый код
Основная проблема наблюдается в файлах контроллеров (например, `ProductController.java`, `OrderController.java`), которые превышают 1500 строк кода каждый. В них содержится значительная часть:
- Бизнес-логики
- Обработки запросов
- Валидации данных
- Операций с базой данных
- Генерации ответов

Проблемы, выявленные:
- Избыточный размер файлов
- Длинные и сложные методы
- Смешение уровней абстракции
- Плохое разделение обязанностей

Это существенно снижает поддерживаемость и повышает риск внесения ошибок при модификациях.

## 1.2 Дублирование кода
Дублирующаяся логика присутствует, особенно в сервисах и контроллерах:
- Похожие реализации CRUD для сущностей (Product, Order, User, Cart)
- Повторяющаяся логика обработки форм и DTO
- Аналогичные блоки try/catch
- Повторяющиеся шаблоны валидации

Отсутствие переиспользуемых абстракций нарушает принцип DRY и усложняет будущие изменения.

## 1.3 Отсутствие автоматизации (Тестирование, Сборка, Развёртывание)
В проекте отсутствуют:
- Юнит-тесты
- Интеграционные тесты
- Автоматизированная тестовая среда
- Правильная конфигурация CI/CD

Фактически, непрерывная интеграция не реализована.

## 1.4 Запутанная архитектура
Архитектура демонстрирует структурные проблемы:
- Бизнес-логика размещена непосредственно в контроллерах
- Отсутствует выделенный слой сервисов для сложной логики
- Крупные монолитные модули
- Частичная ручная валидация без использования аннотаций Jakarta Bean Validation повсеместно

Перегруженные контроллеры являются явным признаком деградации архитектуры.

## 1.5 Потенциально неэффективная реализация
Функциональность корзины и заказов может быть неоптимизирована:
- Нет валидации производительности
- Нет стриминга для генерации больших отчётов или экспортов
- Нет профилирования производительности

Из-за отсутствия тестов и метрик производительности эффективность невозможно объективно проверить.

## 1.6 Незакоммиченный код / Долгоживущие ветки
Эта проблема не выявлена. Практики работы с версионным контролем выглядят приемлемыми.

## 1.7 Отсутствие технической документации
В проекте отсутствуют:
- Файл README с инструкциями по запуску
- Документация архитектуры (включая UML-диаграммы)
- Документация API (Swagger или OpenAPI)
- Описание структуры базы данных
- Документация бизнес-логики

Это существенно усложняет onboarding и поддержку.

## 1.8 Отсутствие тестовой среды
Отсутствуют:
- Юнит-тесты
- Автоматизированные регрессионные тесты
- Настройка тестовой базы данных

В результате любая модификация может внести регрессии, которые останутся незамеченными.

## 1.9 Отсутствие непрерывной интеграции
Отсутствует автоматическое:
- Выполнение тестов
- Валидация качества кода
- Проверка сборки
- Пайплайн статического анализа

Это повышает риск попадания нестабильного кода в основную ветку.

# 2. План снижения технического долга

## 2.1 Рефакторинг архитектуры
- Разделить контроллеры по сущностям (Product, Order, User, Cart) на отдельные файлы/пакеты
- Ввести слой сервисов для бизнес-логики
- Использовать generic-репозитории и аннотации Spring где возможно
- Выделить общую логику в утилитарные классы

Оценка усилий: 16–24 часа

## 2.2 Устранение дублирования кода
- Ввести базовые абстракции для CRUD-операций
- Централизовать логику валидации в DTO с аннотациями
- Унифицировать обработку ошибок с помощью @RestControllerAdvice

Оценка усилий: 8–12 часов

## 2.3 Введение автоматизированного тестирования
- Написать юнит-тесты для валидаторов и сервисов
- Покрыть основные CRUD-операции интеграционными тестами
- Настроить тестовую базу данных (H2 или embedded PostgreSQL)

Оценка усилий: 12–20 часов

## 2.4 Внедрение CI
- Настроить автоматическое выполнение тестов (GitHub Actions или Jenkins)
- Добавить линтинг (Checkstyle, SonarQube)
- Добавить базовую проверку сборки

Оценка усилий: 4–6 часов

## 2.5 Документация
- Создать файл README с инструкциями по запуску и развертыванию
- Документировать архитектуру (с UML-диаграммами)
- Документировать API-эндпоинты (с помощью Springdoc OpenAPI)
- Добавить комментарии к критичным модулям

Оценка усилий: 6–10 часов

## 2.6 Улучшения функциональности корзины/заказов
- Улучшить обработку ошибок
- Добавить логирование (SLF4J с Logback)
- Проверить производительность для больших наборов данных

Оценка усилий: 6–8 часов

# 3. Сравнение: Технический долг vs. Нереализованные фичи

Частично реализованные или отсутствующие фичи:
- Полные CRUD-операции для всех моделей
- Полная логика валидации и безопасности (JWT)
- Фильтрация и поиск по продуктам
- Контроль доступа на основе ролей
- Функциональность отчётов и аналитики

Оценка усилий на доработку фич: 30–40 часов

Оценка усилий на снижение технического долга: 50–70 часов

Объём технического долга сопоставим или даже превышает усилия, необходимые для завершения отсутствующих фич.

# 4. Заключение

Проект находится в активной разработке, но накопленный технический долг уже влияет на:
- Читаемость кода
- Скорость разработки
- Риск дефектов
- Долгосрочную поддерживаемость

Полное устранение технического долга на данном этапе существенно задержит доставку фич. Однако игнорирование его приведёт к дальнейшей деградации архитектуры. Наиболее разумная стратегия — **частичное снижение долга**, включая:
1. Рефакторинг контроллеров и сервисов
2. Введение базового автоматизированного тестирования
3. Постепенное улучшение архитектуры параллельно с разработкой фич

Этот сбалансированный подход минимизирует долгосрочные риски без остановки прогресса проекта.
